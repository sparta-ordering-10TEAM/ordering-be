# 📋 배달 및 주문 관리 플랫폼 (Monolithic Service)

> 본 프로젝트는 음식점 조회부터 주문, 결제, 배송 및 AI 기반 상품 설명 생성까지 유기적으로 연결된 배달 커머스 백엔드 플랫폼입니다. Layered Architecture를 기반으로 견고한 도메인 모델을
> 구축하며, 향후 MSA(Microservices Architecture)로의 확장성을 고려하여 설계되었습니다.

## ⚙️ Core Architecture & System Rules

### 1. 데이터 거버넌스 및 감사 (Data Governance & Audit)

* **Soft Delete:** 모든 데이터는 물리적 삭제를 금지하며, `deleted_at` / `is_deleted` 필드를 통한 논리 삭제(Soft Delete) 방식으로 관리합니다.
* **상품 은닉 분리:** 상품의 '숨김 처리'와 '삭제 처리'는 상호 성격이 다른 비즈니스 요구사항이므로 별도의 물리 필드로 엄격히 분리합니다.
* **글로벌 데이터 감사:** 수정/삭제가 일어나지 않는 단방향 로그성 테이블을 제외한 모든 엔티티는 생성일시, 생성자 ID, 수정일시, 수정자 ID, 삭제일시, 삭제자 ID를 추적 레코드로 기록합니다.
* **식별자 정책:** 시스템의 고유 식별자는 예측 불가능성을 위해 **UUID**를 원칙으로 합니다. (단, 시스템 복잡도 완화를 위해 **유저 엔티티는 예외적으로 순차 기본키(Serial) 정책 적용**)
* **테이블 명명 규칙:** 데이터베이스 내 모든 테이블은 명확한 격리를 위해 `p_` 접두사를 사용합니다. (e.g., `p_user`, `p_order`)

### 2. 세밀한 권한 제어 (Role-Based Access Control)

Spring Security 기반 인터셉터를 통해 엔드포인트 및 필드 레벨의 권한을 엄격하게 검증합니다. 토큰 만료 전 권한 변동(e.g., OWNER ➔ CUSTOMER, MANAGER 삭제 등)으로 인한 보안
취약점을 방지하기 위해, **유저 테이블 접근을 최소화하면서 실시간 유효 권한을 검증하는 최적화 로직**을 적용합니다.

* `CUSTOMER`: 본인의 주문 내역 조회 및 취소/리뷰 작성 권한
* `OWNER`: 본인 소유의 가게 정보, 메뉴 관리, 주문 수락/거절 및 배송 단계 변경 권한 (라이더 미할당 가정)
* `MANAGER`: 시스템 내 모든 음식점 및 주문 정보에 대한 모니터링 및 범용 관리 권한
* `MASTER`: 최고 최고 최고 관리자 권한. 시스템 전권을 가지며 `MANAGER` 계정에 대한 생성/수정/삭제 제어

## 📌 Functional Requirements

### 공통 도메인 규칙

* 모든 도메인은 기본적으로 **CRUD 및 조건 검색(Search)**이 구현됩니다.
* 검색 기능은 `@RequestParam`을 활용한 페이징(`Page` 또는 `Slice`) 처리를 따르며, 기본 정렬 기준은 **생성일시 내림차순**입니다.
* 페이지 노출 건수는 **10건, 30건, 50건**을 표준 규격으로 제한하며, 이외의 요청 진입 시 **기본 10건으로 강제 고정**합니다.

### 1. 사용자 (User)

* **회원가입 규칙:**
  * `username`: 4자 이상 10자 이하, 알파벳 소문자와 숫자의 조합
  * `password`: 8자 이상 15자 이하, 알파벳 대소문자, 숫자, 특수문자의 복합 조합
* **로그인 인프라:** 인증 성공 시 유저 메타데이터를 Payload에 포함한 JWT를 발급하며, Header 또는 Body를 통해 클라이언트에 전달합니다. 해당 토큰 내 Payload 정보를 가공하여
  `UserDetails` 객체를 구성합니다.

### 2. 주문 및 상품 (Order & Product)

* **비즈니스 제약:** 온라인 주문만 지원하며, 주문 생성 후 **5분 이내**에만 클라이언트 측에서 주문 취소가 가능합니다.
* **주문 라이프사이클:** `주문요청(CUSTOMER)` ➔ `주문수락/거절(OWNER)` ➔ `조리완료(OWNER)` ➔ `배달중(OWNER)` ➔ `배달/주문완료(OWNER)`
* **AI 상품 설명 자동 생성:** Gemini API 연동을 통해 프롬프트 기반 상품 설명 자동 생성을 지원합니다. 생성 유무 필드가 `true`일 경우 외부 API를 호출하여 데이터 필드를 채우며, 원본
  프롬프트와 AI 응답 본문은 사후 분석을 위해 별도의 별도 AI 로깅 테이블에 보관합니다.

### 3. 결제 및 배송 (Payment & Delivery)

* **결제 정책:** 신용카드 결제 수단만 단일 지원하며, 외부 PG사 연동은 Mocking 처리하여 플랫폼 내부 DB 테이블에 원장(Ledger) 형태로 기록합니다.
* **배송 인프라:** 주문 요청 시 **배송지 주소**는 필수 입력 사항이며, 사장님(`OWNER`)의 조작에 따라 유기적으로 배송 상태가 변경 및 추적됩니다.

### 4. 음식점 관리 (Restaurant)

* **지역 분류 인프라:** 초기 비즈니스는 서울 광화문 반경으로 한정 운영하나, 전국 단위 확장을 전제한 계층형 지역 분류 시스템을 설계합니다. 지역별 필터링 및 메타데이터 동적 제어를 지원합니다.
* **카테고리 팩토리:** 음식점은 한식, 중식, 분식, 치킨, 피자의 5대 카테고리로 엄격히 분류됩니다.

### 5. 리뷰 및 평점 (Review & Rating)

* 배송 및 주문 완료가 최종 확정된 주문 건에 한해 고객이 1~5점 사이의 평점과 리뷰를 남길 수 있습니다.
* 음식점 목록 조회 시 발생할 수 있는 N+1 연산 부하를 차단하면서 평점 평균을 효율적으로 노출할 수 있는 아키텍처(e.g., Read Model 분리 또는 주기적 배치 집계 등)를 설계합니다.
